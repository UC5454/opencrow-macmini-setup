## 1. データアクセス設計

### アクセス許可

| データ種別 | 許可 | 具体例 |
|---|:---:|---|
| 公開情報 | ✅ | Web、SNS、ニュース、論文、GitHub公開リポジトリ |
| 自社の公開コンテンツ | ✅ | ブログ、Note記事、公開スライド、公開動画 |
| 自社コードベース | ✅ | 開発中のコード（機密ロジック除く） |
| 非機密の社内ドキュメント | ✅ | マニュアル、ガイドライン、テンプレート |
| カレンダー（タイトルのみ） | ✅ | 予定の有無・時間帯（詳細除く） |
| 本人が直接入力した情報 | ✅ | メモ、アイデア、タスク、チェックイン |

### アクセス禁止

| データ種別 | 禁止 | 理由 |
|---|:---:|---|
| 顧客データ全般 | ❌ | 個人情報保護・信頼性 |
| 財務データ | ❌ | P/L、B/S、売上、利益率、コスト |
| 人事・給与 | ❌ | 給与、評価、個人情報 |
| 契約・NDA | ❌ | 契約書、秘密保持、取引条件 |
| 経営戦略文書 | ❌ | 中期計画、M&A検討、未公開方針 |
| CRM | ❌ | 顧客レコード、商談記録 |
| 認証情報 | ❌ | APIキー、パスワード（LiteLLM経由のみ） |

## 2. コンテンツ制作・マーケティング（8件）

公開情報と自社コンテンツだけで完結する。最も制約の影響が少ない領域。

### 2-1. Note/ブログ記事パイプライン

| 項目 | 内容 |
|---|---|
| **概要** | テーマ設定→公開情報リサーチ→記事執筆→サムネイル画像生成→品質レビューまでを全自動で実行 |
| **データソース** | 公開情報（Web記事、論文、SNS投稿）のみ |
| **実行頻度** | テーマ投入時（オンデマンド）、または週次で自動テーマ提案 |
| **エージェント構成** | リサーチャー → ライター → デザイナー → レビュアー（4体並列可能） |
| **期待効果** | 1記事あたり3-5時間の工数削減。月10-20記事の安定量産 |

**実行イメージ:**
```
[ユーザー] "AIエージェントの業務活用"で記事書いて
    ↓
[リサーチャー] Web検索 → 事例収集 → research.md 出力
    ↓
[ライター] research.md → 記事構成 → article.md 出力
    ↓ （並列実行）
[デザイナー] article.md → サムネイル画像生成
[レビュアー] article.md × research.md → 事実確認 → 修正指示
    ↓
[完成] article.md + thumbnail.png → Slack通知
```

## 4. 開発・DevOps（6件）

自社コードベースのみで完結。機密ロジックを含むリポジトリは対象外とする。

### 4-1. 自動PRレビュー

| 項目 | 内容 |
|---|---|
| **概要** | 新しいPRが作成されたら自動でdiffを解析し、バグ・スタイル違反・セキュリティ問題を検出してコメント投稿 |
| **データソース** | 対象リポジトリのコードのみ |
| **実行トリガー** | PR作成・更新時（GitHub Webhook） |
| **検出項目** | バグパターン、スタイル違反、セキュリティ脆弱性、パフォーマンス問題 |
| **期待効果** | レビュー待ち時間の短縮。人間は設計・ロジックに集中 |

| **データソース** | 対象リポジトリのコードのみ |
| **出力** | テストファイル（ユニットテスト、結合テスト） |
| **期待効果** | テストカバレッジの向上。繰り返しテストはAI、エッジケースは人間 |

### 4-3. ドキュメント自動同期

| 項目 | 内容 |

## 5. 生産性ツール（5件）

| 項目 | 内容 |

| **概要** | 毎朝の定時に「天気 + 今日の予定数 + 業界ニューストップ5」を統合配信 |

| **配信先** | Element / Telegram / Slack |
| **期待効果** | 朝の情報収集ルーティンを30分以上短縮 |

**出力例:**
```
☀️ 仙台 晴れ 18℃/8℃

  - 14:00 クライアント打合せ
  - 16:00 開発レビュー

📰 今日のAIニュース TOP5
  1. OpenAI、新モデル発表（TechCrunch）
  2. EU AI Act施行日が確定（Reuters）
  3. ...
```

## 6. マルチエージェント基盤（4件）

### 6-1. Agent Squad（4-5体並列稼働）

| 項目 | 内容 |
|---|---|
| **概要** | 各エージェントに明確な役割を割り振り、24/7で並列稼働させる |
| **推奨構成** | 下記参照 |
| **通信** | Element（Matrix）でチャンネル別に管理 |

**推奨エージェント構成（5体）:**

| エージェント名 | 役割 | 主な担当ユースケース |
|---|---|---|
| **Scout** | リサーチ・情報収集 | 3-1〜3-7（ニュース、競合、トレンド、法規制） |
| **Scribe** | コンテンツ制作 | 2-1〜2-6（記事、SNS、ニュースレター、動画企画） |
| **Forge** | 開発・DevOps | 4-1〜4-5（PRレビュー、テスト、デプロイ） |
| **Pulse** | 監視・通知 | 5-1, 5-5, 6-4（ブリーフィング、スケジュール、ヘルスチェック） |
| **Nexus** | 統括・連携 | 6-2（エージェント間タスク受け渡し、進捗管理） |

---

### 6-2. Agent-to-Agent連携

| 項目 | 内容 |
|---|---|
| **概要** | エージェント間でタスクを自動ハンドオフ。例: Scout→Scribe（リサーチ結果→記事化）、Forge→Pulse（デプロイ完了→通知） |
| **通信方式** | 共有ファイル（STATE.yaml等）、またはMatrix内メッセージ |
| **パイプライン例** | Scout: トレンド検出 → Scribe: 記事ドラフト作成 → Nexus: 品質チェック → Scribe: SNS投稿文生成 |

### 6-3. Mission Controlダッシュボード

---

### 6-4. Self-Healing監視
